# AGENTS.md

이 문서는 이 저장소에서 Codex(에이전트)와 협업할 때의 기본 작업 규칙을 정의한다.

## 1) 코딩 스타일
- 기존 코드베이스의 스타일과 구조를 우선 따른다.
- 변경 범위는 요청된 기능/버그에 필요한 최소 단위로 유지한다.
- 의미 있는 변수명/함수명을 사용하고, 불필요한 주석은 추가하지 않는다.
- 복잡한 로직에는 짧고 명확한 설명 주석만 추가한다.
- 가능하면 기존 유틸/모듈을 재사용하고 중복 구현을 피한다.

## 2) 브랜치/커밋 규칙
- 기본 작업 브랜치는 `main` 기준으로 진행한다. (별도 지시가 있으면 해당 브랜치 사용)
- 커밋 메시지는 `type: summary` 형식을 사용한다.
- 예시 타입: `feat`, `fix`, `docs`, `refactor`, `chore`, `test`
- 커밋은 논리적으로 의미 있는 단위로 나눈다.
- 사용자 지시가 없으면 강제 푸시(`--force`)와 히스토리 재작성은 하지 않는다.

## 3) 테스트/배포 순서
- 코드 변경 후 가능한 범위에서 관련 테스트/검증을 우선 수행한다.
- 테스트가 불가능한 환경이면 그 사실과 이유를 명시한다.
- 배포 관련 변경은 영향 범위와 롤백 포인트를 함께 기록한다.
- 사용자 요청이 있으면 수정 -> 검증 -> 커밋 -> 푸시 순서로 진행한다.

## 4) 협업 규칙 (중요)
- 위험 작업(삭제/리셋/권한승인 필요 작업)만 사용자에게 확인한다.
- 그 외 작업은 중간 확인 없이 바로 진행한다.
- 일반 작업 기본 흐름: 수정 -> 커밋 -> 푸시
- 파괴적 명령 예시: `rm`, `git reset --hard`, 강제 체크아웃/강제 푸시

## 5) 문서 작업 규칙
- README/문서는 처음 보는 사람이 이해할 수 있게 목적, 구성, 실행 방법 순서로 작성한다.
- 아키텍처 다이어그램은 가능하면 Mermaid로 작성하고, 렌더링 오류가 없도록 단순 명료하게 유지한다.
